非常好的问题 👏——这是很多做 Spring Boot 工具类开发时都会遇到的场景。
你说得没错：工具类确实更适合写静态方法，但是你又想使用 RedisTemplate 这种由 Spring 管理的 Bean。

✅ 问题本质

Spring 的 @Autowired 只会注入到 被 Spring 容器托管的对象的实例字段。
静态方法属于类本身（不是对象），所以它不受 Spring 容器管理，无法直接注入依赖。

✅ 正确解决方案：使用 “静态 setter 注入”

这种方式最常用、最安全。

✅ 使用方式

只要你的 Spring 容器启动成功（比如项目已经启动，RedisTemplate 已经注入），
你就可以在任何地方（包括 Controller、Service 甚至别的工具类）里直接调用
ScrollUtil.scrollGetIds(...)。

✅ 这种方式为什么安全？

  • Spring 在项目启动时会实例化 ScrollUtil；
  • 调用 setRedisTemplate(...) 自动注入 Bean；
  • 静态字段在类加载时就赋值完毕；
  • 后续静态方法可直接使用该 redisTemplate。

🔥 非常棒的问题 —— 这其实问到了 Spring 注入机制的底层原理。
很多人都知道 “@Autowired 不能注入静态字段”，
但不知道 为什么“不能注入字段，却能注入到 setter 方法中给静态变量赋值”。
我来帮你彻底讲清楚👇

✅ 一、Spring 的依赖注入（DI）本质

Spring 的自动注入（@Autowired）主要有两种常见形式：

  1. 字段注入（Field Injection）

    👉 Spring 会通过反射（Field.set()）
    把容器里的 Bean 注入到这个实例字段上。

  2. 方法注入（Setter / 构造器 Injection）

    👉 Spring 调用这个方法，并把容器中找到的 Bean 作为参数传进去。

✅ 二、为什么不能注入 static 字段？

因为 Spring 的依赖注入发生在 Bean 实例创建后。
而 static 字段属于 类本身，不是属于实例的。

举个例子：

Spring 创建 ScrollUtil 实例时，会尝试通过反射给字段赋值。
但由于这个字段是静态的（static），它不属于实例对象，Spring 在反射时会这样：

👉 实际效果：虽然赋值成功，但这个静态字段并不会被真正注入，因为 Spring 的生命周期是按实例级别管理的，静态变量不会被容器跟踪、销毁、管理。

所以 Spring 默认跳过静态字段的依赖注入（框架设计层面直接忽略掉）。

✅ 四、核心区别总结

注入方式 Spring 自动赋值目标 能否作用于 static 字段 原理
字段注入 实例字段 ❌ 不行 直接反射设置实例属性
setter 方法注入 方法参数（你自己决定赋值逻辑） ✅ 可以（手动赋值） Spring 调用方法时传入 Bean

✅ 五、通俗一点打个比方

假设 Spring 是一个“送快递”的人，
Bean 是“包裹”，
你的类是“收件人”。

  • 字段注入：Spring 直接往你的“信箱”（实例属性）里塞包裹。
    → 静态字段没有信箱，Spring 找不到地方放。
  • setter 注入：Spring 把包裹交到你手上（方法参数），
    你要放哪儿，是你自己的事。
    → 你可以自己塞进全局仓库（静态字段）也行。

✅ 六、结论

@Autowired 不能直接注入静态字段，
但你可以用一个被 Spring 调用的实例方法，把容器里的 Bean 手动赋值给静态字段。

也就是说：

  • 字段注入 → Spring 直接赋值，不支持 static
  • setter 注入 → 你手动赋值，Spring 只是负责“调用方法 + 传参数”

工具类（Utility Class）的本质应该是 无状态、无依赖、可独立使用的。

如果一个工具类必须依赖 Spring 容器的 Bean（比如 RedisTemplate），
它就不再是“纯工具类”，而是“业务辅助组件”。
这种情况下放进容器里是 可以的，但不是必须的。

⚠️ 五、什么时候真的该放进容器？

只有当这个工具类：

  • 内部要用多个 Bean（比如 RedisTemplate、RabbitTemplate、DataSource 等）；
  • 需要事务支持（@Transactional）；
  • 需要使用 Spring 配置（比如 @Value("${xxx}")）；
    才适合 @Component 交给容器托管。

否则，就不值得那样做。
